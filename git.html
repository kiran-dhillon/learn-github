Git & GitHub Notes — Developer Cheat Sheet

Short, practical notes and commands you can copy. Simple steps for daily developer work (Kiranjan).

Basics Workflow Branching Merge Conflicts Cheatsheet Tasks

1. Basics — What & Why

Git = distributed version control. GitHub = cloud hosting + collaboration platform. Use Git locally to track snapshots; push to GitHub for sharing & backup.

Common flow (local)

  1. Make changes in working directory
  2. git add → stage changes
  3. git commit → commit snapshot
  4. git push → upload to remote

First-time commands

# set identity (only once per machine)
$ git config --global user.name "Your Name"
$ git config --global user.email "you@example.com"

# create repo locally
$ git init

# create repo on GitHub then connect remote (example)
$ git remote add origin https://github.com/username/repo.git
$ git branch -M main
$ git push -u origin main

2. Developer Workflow (Real projects)

Typical pattern used in teams:

  • Clone repo: git clone <url>
  • Create feature branch: git checkout -b feature/xyz
  • Work locally — add & commit frequently (small commits)
  • Push feature branch: git push origin feature/xyz
  • Open Pull Request (PR) on GitHub → review → merge
  • Delete branch after merge

Important commands

$ git status          # current state
$ git add .
$ git commit -m "short, meaningful message"
$ git push origin feature/xyz
$ git pull origin main   # bring latest main locally

3. Branching & Names

Use readable branch names: feature/login, fix/header-bug, hotfix/critical-1.

Create & switch

$ git branch new-branch      # create
$ git switch new-branch      # switch (recommended)
$ git checkout -b new-branch # create + switch

Merge vs Rebase

Merge creates a merge commit and preserves history. Rebase rewrites commits to create a linear history (use carefully on shared branches).

# merge feature into main
$ git switch main
$ git merge feature/xyz

# rebase feature onto latest main (local)
$ git switch feature/xyz
$ git rebase main

4. Merge Conflicts (How to solve)

Conflict happens when same lines changed in two branches. Steps:

  3. Keep correct code, remove markers

  5. git commit (or git rebase --continue if rebasing)

5. Pull Requests & Code Review

  • Open PR from your branch → target main (or develop)
  • Write clear PR description: what changed, why, screenshots, steps to test
  • Request reviewers, address feedback with additional commits
  • Prefer small PRs — easier to review

6. Cheatsheet — Most used commands

# clone remote repo
$ git clone https://github.com/username/repo.git

# check status & history
$ git status
$ git log --oneline --graph --decorate

# stage & commit
$ git add path/file.js
$ git commit -m "fix: header layout"

# push & pull
$ git push origin branch-name
$ git pull origin main

# branches
$ git branch
$ git branch -d branch-name
$ git push origin --delete branch-name

# stash (save WIP)
$ git stash         # save
$ git stash list
$ git stash apply   # re-apply

# undo local changes
$ git checkout -- file.txt   # restore file from HEAD
$ git reset --hard HEAD      # WARNING: discards local commits and changes

7. Tasks (Practice) — Do these now

  1. Clone any public repo from GitHub.
  2. Create a branch: git checkout -b practice/yourname
  3. Edit README (add a line), commit and push.
  4. Open a Pull Request and merge it (if you own the repo) or create PR to a fork.

8. Tips & Best Practices

  • Commit early, commit often. Small commits are easier to review.
  • Write meaningful commit messages: type(scope): short summary
  • Use .gitignore to avoid committing node_modules, secrets, build artifacts.
  • Never force-push to a shared branch without coordination.
  • Protect main branch with branch-protection rules on GitHub.
